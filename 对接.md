## 0. 必须先统一的基础约定

### 0.0 材料优先级与当前判断

后端 `README.md` 明确：

- 当前正式交付目录是 `backend/`。
- `APP/grade-backend` 仅作为历史参考实现。
- 当前需求基线来自仓库内 `demand.md`。
- 正式接口前缀是 `/api/v1/grade/**` 和 `/internal/api/v1/grade/**`。
- 身份由 Nginx/Gateway 注入 `X-User-ID`、`X-User-Role`、`X-User-Name`。

三份 docx 进一步确认：

- `模块6-技术选型.docx`：业务服务不自行签发/验证 JWT，只读取网关注入身份头。
- `模块6-跨小组数据流动设计.docx`：A/C/D/E 组是各自数据的 SSOT，F 组通过本地快照和影子表支撑高频查询，前端不应跨组编排成绩计算。
- `模块6-数据库设计.docx`：`grade_record` 是分项窄表，`grade_total` 是审批后总评表，学生只查 `published + is_current`，GPA 通过队列异步刷新。

当前需要和后端同学明确一个原则：

```text
前端联调先按 backend/ 当前代码实现，还是按 demand.md 目标契约等待后端补齐？
```

我的建议：

- 前端代码先按 `backend/` 当前可运行接口做适配，保证主流程能跑。
- `demand.md` 与当前代码的差异列入本文件，作为后端 P0/P1 调整项。
- 后端每完成一个目标契约，再由前端从兼容逻辑切换到目标逻辑。

### 0.1 API 前缀

后端当前实际前缀是：

```text
/api/v1/grade
```

前端将按该前缀实现。请后端确认网关、Nginx、测试环境部署后是否仍保持这个前缀。

需要后端确认：

- 测试环境完整 base URL 是什么。
- 是否会经过统一网关。
- 是否存在 `/api/grade/v1` 这种旧前缀；如果不存在，请后端文档同步删除。

目前判断：

- `README.md`、`技术选型.docx`、`backend/app/api/v1/router.py` 三者都支持 `/api/v1/grade`，这里已基本确定。

### 0.2 响应结构

后端当前返回：

```json
{
  "code": 10000,
  "message": "Success",
  "data": {}
}
```

前端模板默认识别的是：

```json
{
  "code": 200,
  "msg": "成功",
  "data": {}
}
```

前端会在成绩模块做局部适配，但需要后端确认：

- `10000` 是否是唯一成功业务码。
- 所有非 `10000` 是否都视为业务失败。
- 业务失败时是否一定返回 `message`。
- 后端当前部分代码存在 `APIResponse.error(None, "...")`，可能返回 `code: null`，建议统一改成明确错误码，例如 `10400` 或 `70103`。

与 `demand.md` 的差异：

- `demand.md` 要求业务失败时也有明确 `code`，且 `data` 通常为 `null`。
- 当前代码部分路径可能返回 `code: null`，这会破坏前端错误码映射。

### 0.3 鉴权与角色

后端当前依赖请求头：

```http
X-User-ID
X-User-Role
X-User-Name
```

前端现有模板只会自动带：

```http
Authorization: Bearer <token>
x-access-token: <token>
```

需要后端确认：

- 联调阶段是否由前端临时传 `X-User-*`。
- 正式环境是否由网关根据 JWT 注入 `X-User-*`。
- 后端是否会继续直接信任浏览器传入的 `X-User-*`，这存在安全风险。

根据 `README.md` 和 `模块6-技术选型.docx`，正式方案应为：

```text
Client -> Nginx/Gateway 校验 JWT -> 注入 X-User-* -> backend
```

前端对接建议：

- 联调直连 FastAPI 时，前端可临时注入 `X-User-*`。
- 走网关后，前端只传 JWT，由网关注入身份头。
- 后端应避免把“浏览器直传 X-User-*”作为正式安全方案。

角色枚举也不一致：

| 前端角色 | 后端当前允许角色 |
| --- | --- |
| `student` | `student` |
| `teacher` | `teacher` |
| `academic_admin` | `admin` / `dean` |

需要后端确认：

- 教务管理员最终角色名是 `admin`、`dean`，还是支持 `academic_admin`。
- 如果后端不支持 `academic_admin`，前端会临时映射为 `admin`。

## 1. 课程与成绩配置接口

### 1.1 教师课程范围过滤缺失

当前接口：

```text
GET /api/v1/grade/courses
```

后端实现从 `LocalRosterShadow` 全量汇总课程，没有按当前教师过滤。

前端影响：

- 教师可能在课程下拉中看到全部课程。
- 权限边界不成立，前端只能做体验层过滤，不能兜底安全。

建议后端调整：

- 根据 `X-User-ID` 过滤教师任课课程。
- 管理员 `admin/dean` 可以看到全部课程。
- 返回中可增加 `teacher_id`、`teacher_name`，方便管理员筛选。

### 1.2 成绩配置缺少 `max_score`

当前配置项只有：

```ts
id
course_id
semester
component_type
component_sub_id
weight
data_source
is_locked
```

前端原计划支持每个分项配置满分 `max_score`，但后端分数校验固定为 0–100。

需要后端确认：

- 本期是否固定所有分项满分 100。
- 如果未来支持不同满分，请补 `max_score` 字段，并在 `GradeRecordCreate.score` 校验中按配置校验。

### 1.3 权重错误返回码

当前 `_save_grade_config` 中权重不为 100 时调用：

```py
APIResponse.error(None, "component weights must add up to 100")
```

建议后端调整：

- 返回 `70103 GRADE_WEIGHT_INVALID`。
- 前端可据此展示“成绩组成权重之和必须为 100%”。

## 2. 成绩录入与查询接口

### 2.1 成绩记录是分项窄表，前端需要聚合

当前：

```text
GET /grade-records
```

返回的是一条分项成绩一行：

```ts
student_id
student_no
course_id
semester
component_config_id
component_type
score
status
```

前端实际页面需要“一名学生一行，多个分项动态列”。

前端可以临时自行 pivot，但有问题：

- `GET /grade-records` 的分页是按分项行分页，前端按学生行展示会错页。
- 同一学生的不同分项可能被分页切开。
- 后端不返回 `student_name`、`major`，前端还要额外请求学生名单合并。

建议后端新增聚合接口：

```text
GET /api/v1/grade/courses/{course_id}/grade-sheet?semester=2026-1&page=1&page_size=20
```

建议返回：

```ts
{
  total: number;          // 学生数，不是分项数
  page: number;
  page_size: number;
  components: GradeComponent[];
  rows: Array<{
    student_id: string;
    student_no: string;
    student_name: string;
    major: string | null;
    total_score?: number | null;
    gpa?: number | null;
    status: string;
    records: Array<{
      id: number | null;
      component_config_id: number;
      score: number | null;
      version_no?: number;
      is_locked: 0 | 1;
      data_source: string;
    }>;
  }>;
}
```

### 2.2 成绩记录缺少课程名、学生名、专业

当前 `GradeRecord` 不返回：

- `course_name`
- `student_name`
- `major`
- `credit`

前端影响：

- 教师查询、管理员查询、导出表格都需要多接口拼接。
- 页面加载复杂且容易出现数据不同步。

建议后端调整：

- 在 `GET /grade-records` 或新聚合接口中 join `LocalRosterShadow`，返回课程名、学生名、专业、学分。

### 2.3 内联编辑建议支持版本冲突

后端 `GradeRecordUpdate` 已有 `version_no` 字段，但当前更新逻辑没有检查版本冲突。

建议后端调整：

- 如果前端传入 `version_no` 与数据库当前版本不一致，返回 `10410 VERSION_CONFLICT`。
- 前端收到后刷新该行并提示“数据已被他人修改，请刷新后重试”。

### 2.4 手工录入与外部来源限制

后端当前逻辑：

- `data_source != manual` 的成绩不能手工编辑。
- 只能通过 `/import-exam` 或 `/import-forum` 写入。

前端会据此禁用非手工列。

需要后端确认：

- `data_source` 的最终枚举是否固定为 `manual/exam/forum`。
- 是否还有其他来源，例如 `experiment`、`attendance`。

## 3. Excel 与外部成绩导入

### 3.1 Excel 导入上下文来源不适合前端弹窗

当前接口：

```text
POST /grade-records/import-excel
```

只接收 `file`，并要求 Excel 表头包含：

```text
student_id, course_id, semester, component_config_id, score
```

前端原计划是用户在弹窗里选择课程、学期、目标分项，Excel 只包含学生和分数。

前端影响：

- 用户需要在 Excel 每行重复填写课程、学期、分项 ID，体验差。
- `component_config_id` 是数据库 ID，普通教师很难知道。

建议后端按 `demand.md` 调整：

```text
POST /grade-records/import-excel
multipart/form-data:
  file
  course_id
  semester
  component_type
  component_sub_id?
```

Excel 模板只需要：

```text
student_no 或 student_id, score, result_type?
```

另建议新增：

```text
GET /grade-records/import-template?course_id=&semester=&component_config_id=
```

用于下载模板。

### 3.2 在线测试导入职责不清

当前接口：

```text
POST /grade-records/import-exam
```

要求前端传：

```ts
{
  course_id: string;
  semester: string;
  scores: Array<{
    student_id: string;
    external_id: string;
    external_name?: string;
    score: number;
  }>;
}
```

并且还要求先调用：

```text
POST /grade-records/external-mappings
```

前端影响：

- 前端需要直接对接在线测试组获取 scores。
- 前端还要理解 external mapping，跨组编排复杂。

建议后端调整：

由成绩服务内部调用在线测试服务，前端只传 `demand.md` 中的目标字段：

```ts
{
  course_id: string;
  semester: string;
  component_type: string;
  component_sub_id?: string;
}
```

后端负责：

- 找在线测试成绩。
- 匹配学生。
- 按 `external_component_mapping` 或 `component_type/component_sub_id` 找到本地成绩组成项。
- 写入 `GradeRecord`。
- 返回成功/失败明细。

### 3.3 论坛成绩导入本期是否开放

后端已有：

```text
POST /grade-records/import-forum
```

但前端当前计划是后续能力。

需要后端确认：

- 本期验收是否要求论坛成绩导入。
- 如果不要求，前端会隐藏或置灰该入口。

## 4. 成绩计算、审批与发布流程

### 4.1 `calculatePreview` 不是真正预览

当前接口：

```text
POST /courses/{course_id}/grade-calculations
```

只返回：

```ts
{
  course_id: string;
  semester: string;
  calculated_count: number;
}
```

它不会返回每个学生的总评和 GPA，也不落表。

前端影响：

- 无法实现“计算总评并预览综合成绩 + 绩点”。

建议后端调整返回：

```ts
{
  course_id: string;
  semester: string;
  calculated_count: number;
  missing_scores: string[];
  rows: Array<{
    student_id: string;
    student_no: string;
    student_name: string;
    total_score: number | null;
    gpa: number | null;
    components: Array<{
      component_config_id: number;
      score: number | null;
      weight: number;
    }>;
  }>;
}
```

### 4.2 审批通过后学生仍看不到成绩

当前流程：

1. 教师提交：`GradeSubmission.status = pending`，`GradeRecord.status = submitted`
2. 管理员通过：`GradeSubmission.status = approved`，`GradeTotal.status = approved`
3. 管理员发布：`GradeTotal.status = published`
4. 学生查询：只查 `GradeTotal.status == published`

前端必须增加“发布成绩”按钮，否则学生端无法查到。

需要后端确认：

- 本期是否要求“审批通过”和“发布”分两步。
- 如果分两步，`GET /grade-submissions` 是否应该返回可发布状态。
- `PUT /grade-submissions/{id}/publish` 是否应同步把 `GradeSubmission.status` 也更新为 `published`，当前代码只更新 `GradeTotal`。

### 4.3 改分通过后是否自动发布

当前改分审批通过后：

- 更新分项 `GradeRecord.score`
- 新建 `GradeTotal(status="approved")`
- 不自动设为 `published`

前端影响：

- 学生端可能看不到改分后的新成绩。
- 管理员还需要一个“发布改分结果”的流程，但后端没有明确接口。

需要后端确认：

- 改分通过后是否应自动发布。
- 如果不自动发布，是否复用 `publishTotalsLegacy({ course_id, semester })`。
- `GradeTotal.status` 是否需要区分 `corrected` / `published`。

### 4.4 改分申请允许状态与前端需求不一致

后端当前只允许：

```py
record.status == "approved"
record.is_locked == 0
```

前端原计划希望对已提交/已发布/已更正成绩申请修改。

需要后端确认：

- 教师应对 `GradeRecord` 的哪个状态发起改分。
- 如果成绩已经发布，`GradeRecord.status` 是否仍保持 `approved`。
- 是否需要在查询接口里返回“可申请改分”的布尔字段，避免前端猜状态。

## 5. 学生查分、学分与统计

### 5.1 学生查分缺少分项明细

当前：

```text
GET /students/me/grades
```

只返回总评：

```ts
course_id
semester
total_score
gpa
credit
```

前端原计划支持展开查看分项明细。

建议后端新增：

```text
GET /students/me/grades/{course_id}/records?semester=2026-1
```

或直接在 `me/grades` 中返回：

```ts
components: Array<{
  component_type: string;

### 5.2 学生查分缺少课程展示字段

- `course_name`

course_name: string;
grade_level?: string;
assessment_type?: string;
```

### 5.3 学分进展接口数据不足

```text
GET /students/me/credits
```

返回：

前端原计划需要：

- 总应修学分
- 已修学分
- 剩余学分
- 按类别统计：必修、专业选修、公共课等

## 6. 课程成绩分析与导出

### 6.1 缺少排名列表

当前：

```text
GET /courses/{course_id}/analysis
```

返回：

- 平均分
- 最高分
- 最低分
- 中位数
- 通过率
- 优秀率
- 分数段分布
- 等级分布
- 排名阈值摘要

但不返回学生排名列表。

前端原计划有“排名表”。

建议后端补：

```ts
rankings: Array<{
  rank: number;
  student_id: string;
  student_no: string;
  student_name: string;
  total_score: number;
  gpa: number;
}>
```

或者提供独立分页接口：

```text
GET /courses/{course_id}/analysis/rankings?semester=&page=&page_size=
```

### 6.2 导出格式不一致

当前：

```text
GET /courses/{course_id}/analysis/export?semester=&format=xlsx
```

只支持 `xlsx`，且返回文件流。

前端原计划有 PDF 报告或 `download_url`。

需要后端确认：

- 本期验收是否要求 PDF。
- 如果只要求 Excel，前端会按 blob 下载实现。
- 如果要求 PDF，请后端支持 `format=pdf`，或确认由前端自渲染。

### 6.3 成绩记录导出也是文件流

当前：

```text
GET /grade-records/export?course_id=&semester=
```

直接返回 xlsx 文件流。

前端会按 blob 下载，不再期待：

```ts
{ download_url: string }
```

需要后端确认：

- 文件名是否通过 `Content-Disposition` 提供。
- 是否需要导出水印、导出人、导出时间、版本号。如果需要，建议后端统一生成。

## 8. 前端可先适配的部分

这些点前端可以先做，不阻塞后端：

- 固定接口前缀 `/api/v1/grade`。
- 成绩模块局部适配 `10000/message`。
- 联调阶段临时注入 `X-User-*` header。
- 将 `academic_admin` 映射为 `admin`。
- 成绩录入页前端 pivot 分项窄表。
- 非 `manual` 分项列置灰。
- 审批页增加“发布”按钮。
- 学分进展、排名表、学生分项明细等后端暂不支持能力先显示空状态或隐藏。
- 导出先按 xlsx blob 下载。

## 9. 希望后端优先调整的 P0 项

1. 统一鉴权方案：确认 `X-User-*` 是临时方案还是网关注入方案。
2. 统一角色枚举：支持或映射 `academic_admin`。
3. 修复 `APIResponse.error(None, ...)`，所有失败返回明确 `code`。
4. 教师课程、成绩、提交、改分接口按当前教师过滤。
5. 明确审批通过后是否必须单独发布；如果必须，完善 `GradeSubmission.status` 的 published 状态。
6. 提供学生行聚合的成绩单接口，避免前端按分项分页导致表格错乱。
7. 学生查分补分项明细或新增安全的学生分项接口。
8. Excel 导入支持表单传 `course_id/semester/component_config_id`，不要要求教师在 Excel 中填写数据库 ID。
9. 在线测试导入由成绩服务内部拉取成绩，避免前端跨组编排。
10. 明确导出格式：只做 xlsx，还是需要 PDF/水印。

## 10. 建议会议议程

建议和后端同学开 30–45 分钟对齐会，按以下顺序过：

1. 基础协议：前缀、响应、鉴权、角色。
2. 核心闭环：录入、提交、审批、发布、学生查分。
3. 数据结构：成绩明细窄表是否新增聚合接口。
4. 导入导出：Excel 模板、在线测试导入、导出格式。
5. 后续增强：学分分类、排名、统计、日志分页。

会议结束需要产出：

- 最终 API 前缀和鉴权方式。
- 后端 P0 调整项负责人和完成时间。
- 前端可以先按现有接口开发的范围。
- 暂不实现的验收项清单。
