## 5. 已知 bug / 风险

这一部分是接手同学优先要处理的内容。

## 5.1 `X-User-ID` 当前传的是 token，不是真实 user_id

文件：

- `src/api/modules/score.ts`

当前实现中：

- `X-User-ID` 使用的是 `userStore.token`

这会导致联调时出现身份字段不准确的问题。后端接口和测试里期望的是类似：

- `T001`
- `A001`
- `S001`

这种真实用户 ID，而不是 `token-teacher` 这种 token 字符串。

**影响：**

- 鉴权字段可能不符合后端预期
- 审批人 / 申请人 ID 可能写错
- 审计日志可能被污染

---

## 5.2 改分申请和审批里也把 token 当用户 ID 用了

文件：

- `src/views/stss/score/change-request/index.vue`
- `src/views/stss/score/change-approval/index.vue`

当前：

- `applicant_id`
- `reviewer_id`

都来自 `userStore.token`。

这是明确 bug，建议优先修复。

---

## 5.3 多个页面的课程选择对“跨学期同课程”不安全

涉及文件：

- `src/views/stss/score/entry/index.vue`
- `src/views/stss/score/query/index.vue`
- `src/views/stss/score/change-request/index.vue`
- `src/views/stss/score/course-analytics/index.vue`

现状：

- 下拉展示使用 `course_id + semester`
- 但选中值只存 `course_id`
- 后续再通过 `find(course => course.course_id === selectedCourseId)` 去反推学期

这会导致：

- 同一课程跨多个学期时，可能命中错误学期
- 查询、审批、导出可能使用错误数据

建议后续改成：

- 课程和学期分开强绑定
- 或直接使用 `(course_id, semester)` 联合作为选择值
